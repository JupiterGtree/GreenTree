## Outcome

Phase 1 replaces the production-risk market, transaction and transparency mocks with a typed server-side data layer. Failed sources now produce explicit unavailable or partial-source states instead of plausible values. UI structure, routes, wallet confirmation and page design were preserved.

This phase deliberately stops before mission administration, Squads account decoding, roadmap administration and larger backend work.

## Verified Phase 1 data

| Data | Production source | Validation |
|---|---|---|
| Network | Solana `getGenesisHash` | Must equal the Solana Mainnet genesis hash |
| Token program and initialized state | Solana `getAccountInfo`, `jsonParsed` | Must be a parsed mint owned by the Classic SPL Token Program |
| Decimals and raw/current supply | Solana `getAccountInfo` + `getTokenSupply` | Both RPC responses must agree; raw amount is validated as an integer string |
| Mint and Freeze Authority | Parsed mint account | Authority is either a valid Solana address or `null` |
| Fixed-supply state | Derived from revoked Mint Authority and current supply | Compared with the owner-published 1B maximum; no static substitution on RPC failure |
| Pool identity and pair | Meteora pool endpoint | Pool, GTREE mint and wrapped SOL mint must match confirmed identifiers |
| GTREE/SOL and SOL/USD | Meteora pool endpoint | Both must be finite and positive |
| GTREE/USD | Derived calculation | Only calculated when both market operands validate |
| Implied valuation | Market price × on-chain supply | Unavailable when on-chain supply cannot be verified |
| Liquidity, volume and fees | Meteora pool endpoint | Missing fields remain unavailable; an explicit reserve-derived liquidity estimate may be shown |
| Quote and minimum received | Jupiter quote endpoint | Exact mints, input amount, slippage, atomic amounts and every route leg are validated |
| Swap transaction | Jupiter swap endpoint | Quote is rebuilt server-side; user wallet must be on-curve; no auto-sign or auto-submit |
| Wallet balances | Solana `getBalance` + `getTokenAccountsByOwner` | Raw lamports/SPL integer strings are retained and token raw amounts use `bigint` |
| Recent transactions | Solana `getSignaturesForAddress` + `getTransaction` | Only confirmed mint/pool activity; labels remain neutral unless logs/balance deltas support them |
| Transparency records | Live token state + official document registry | No fabricated signatures, dates, amounts, reports or incident claims |

## Central data architecture

The production layer is organized around:

- `src/config/data-sources.ts` — authoritative source registry and policies.
- `src/config/server-env.ts` — server-only external endpoint configuration.
- `src/services/http/fetch-json.ts` — timeout, bounded retry and safe structured logging.
- `src/services/solana/rpc.ts` — normalized Solana JSON-RPC calls.
- `src/services/meteora/pool.ts` — validated pool normalization.
- `src/lib/market/jupiter.ts` — validated quote and user-signed swap preparation.
- `src/data/token/get-token-state.ts` — cached normalized token state.
- `src/data/market/get-market-snapshot.ts` — cached normalized market snapshot.
- `src/data/transactions/get-recent-transactions.ts` — cached neutral transaction normalization.
- `src/data/transparency/get-transparency-records.ts` — live/static transparency record composition.
- `src/types/data.ts`, `token.ts`, `market.ts`, `transaction.ts` and `transparency.ts` — application-facing types.

UI components consume normalized data and do not call Solana, Jupiter or Meteora directly. Client components call controlled same-origin route handlers where interaction requires it.

## Cache and refresh policy

| Data | Policy |
|---|---|
| Token state and authorities | Next server cache, 1 hour |
| Market snapshot | Next server cache, 15 seconds |
| Recent transactions | Next server cache, 45 seconds |
| Transparency records | Next server cache, 1 hour |
| Historical candles route | Shared cache header, 30 seconds with 30-second stale-while-revalidate |
| Jupiter quote | No-store; 15-second local UI freshness window |
| Jupiter swap build | No-store, no retry for the stateful build request |
| Wallet balance | Private no-store |
| Official project information | Static/build-time source |

Every external request uses an 8-second timeout. Read-only requests have at most one controlled retry. No new polling loop was introduced; the existing quote countdown only updates local expiry display state.

## Security changes and concerns

Implemented:

- The GTREE mint is fixed to the confirmed Mainnet identifier and cannot silently switch via a public environment variable.
- Solana RPC genesis hash is checked against Mainnet before token state is accepted.
- Pool and both token mints are checked on every Meteora snapshot.
- Jupiter routes are checked against the confirmed pool and exact input/output mints.
- Swap preparation rebuilds the route server-side and rejects off-curve wallet addresses.
- Transactions remain user-confirmed in the connected wallet; the website never signs or submits automatically.
- Raw SPL values use integer strings/`bigint`; external responses are validated before display.
- Private API credentials stay server-only; logs omit headers and payloads.

Open concerns:

- A dedicated rate-limited and monitored Solana Mainnet RPC is required before launch.
- There is no durable stale-data store or deployment-level observability yet.
- The public Meteora data API provides market-derived SOL/USD, not a dedicated oracle proof.
- Meteora's `is_blacklisted` semantics were not found to support a user-facing risk claim, so the value is not shown as one.
- Squads treasury configuration is project-published but not yet independently decoded in this phase.
